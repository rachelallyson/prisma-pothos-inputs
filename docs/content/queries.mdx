# Queries

This page shows how to wire **queries** using the refs from `registerPothosTypes(builder)`. With `--prisma-client-path` set, those refs are typed as Prisma inputs so you can pass args straight to the client.

The [example schema](https://github.com/rachelallyson/prisma-pothos-inputs/blob/main/examples/pothos/src/schema.ts) implements both patterns below.

## Which refs to use

| Operation   | Args | Generated ref(s) |
|------------|------|-------------------|
| **findUnique** | `where` (required) | `refs.ModelWhereUniqueInputType` |
| **findMany**   | `input` (optional: where, orderBy, skip, take) | `refs.ModelFindManyArgs` |

---

## findUnique

Single record by unique criteria. Use `t.prismaField` and pass the Pothos `query` through so relations/select work.

```ts
user: t.prismaField({
  type: 'User',
  args: {
    where: t.arg({ type: refs.UserWhereUniqueInputType, required: true }),
  },
  resolve: (query, _root, args) =>
    prisma.user.findUnique({
      ...query,
      where: args.where,
    }),
}),
```

---

## findMany

List with optional filters, ordering, and pagination. Use `refs.ModelFindManyArgs` and spread into findMany ‚Äî it type-checks when your schema‚Äôs PrismaTypes and context client use the same types (e.g. same Prisma schema, no `$extends`):

```ts
users: t.prismaField({
  type: ['User'],
  args: {
    input: t.arg({ type: refs.UserFindManyArgs, required: false }),
  },
  resolve: (query, _root, args, ctx) =>
    ctx.prisma.user.findMany({ ...query, ...(args.input ?? {}) }),
}),
```

All fields on `FindManyArgs` are optional. If you don't care about pagination, the client can omit `skip` and `take` and send only `where` and/or `orderBy` (or an empty object).

### findMany without pagination (where + orderBy only)

If you want a list field that does **not** expose `skip`/`take` at all, use `WhereInput` and `OrderByInput` as separate args instead of `FindManyArgs`:

```ts
users: t.prismaField({
  type: ['User'],
  args: {
    where: t.arg({ type: refs.UserWhereInputType, required: false }),
    orderBy: t.arg({ type: refs.UserOrderByInput, required: false }),
  },
  resolve: (query, _root, args) =>
    prisma.user.findMany({
      ...query,
      ...(args.where != null && { where: args.where }),
      ...(args.orderBy != null && { orderBy: args.orderBy }),
    }),
}),
```

This keeps the same filtering and ordering capabilities without exposing pagination in the API.

### TypeScript: query type and findMany

When your schema‚Äôs **PrismaTypes** and the **Prisma client** in context use the same types (same schema, no `$extends`), `{ ...query, ...(args.input ?? {}) }` type-checks with no assertion.

**If you use Prisma client extensions (`$extends`):** the extended client‚Äôs types can be out of sync with default PrismaTypes, so you may see ‚ÄúExcessive stack depth.‚Äù Two options:

1. **Generate PrismaTypes from your extended client** (recommended) so the plugin types `query` with the extended client‚Äôs Select/Include and the spread type-checks with no cast. Run the generator with `--output-prisma-types` and `--extended-prisma-type-path` (path to the module that exports your extended client type). That module must also export: `export type ExtendedPrisma = typeof yourExtendedClient;` Then use the generated file as your SchemaBuilder‚Äôs `PrismaTypes`:

   ```bash
   npx @rachelallyson/prisma-pothos-inputs --schema prisma/schema.prisma \
     --output-prisma-types src/__generated__/pothos-prisma-types.ts \
     --extended-prisma-type-path ../db.js \
     --prisma-client-path ../../generated/prisma/client.js
   ```
   (Use a path to the extended-client module relative to the generated file; e.g. `../db.js` when the output is in `src/__generated__/`.)

   In your schema: `import type PrismaTypes from './__generated__/pothos-prisma-types.js';` and pass `PrismaTypes` to the builder. Use the same extended client in context. Then `ctx.prisma.user.findMany({ ...query, ...(args.input ?? {}) })` type-checks.

2. **Cast when spreading:** use the generated `UserFindManyQueryShape` type or `...(query as object)`. See [prisma/prisma#27059](https://github.com/prisma/prisma/issues/27059).
