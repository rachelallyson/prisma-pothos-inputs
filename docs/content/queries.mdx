# Queries

This page shows how to wire **queries** using the refs from `registerPothosTypes(builder)`. With `--prisma-client-path` set, those refs are typed as Prisma inputs so you can pass args straight to the client.

The [example schema](https://github.com/rachelallyson/prisma-pothos-inputs/blob/main/examples/pothos/src/schema.ts) implements both patterns below.

## Which refs to use

| Operation   | Args | Generated ref(s) |
|------------|------|-------------------|
| **findUnique** | `where` (required) | `refs.ModelWhereUniqueInputType` |
| **findMany**   | `input` (optional: where, orderBy, skip, take) | `refs.ModelFindManyArgs` |

---

## findUnique

Single record by unique criteria. Use `t.prismaField` and pass the Pothos `query` through so relations/select work.

```ts
user: t.prismaField({
  type: 'User',
  args: {
    where: t.arg({ type: refs.UserWhereUniqueInputType, required: true }),
  },
  resolve: (query, _root, args) =>
    prisma.user.findUnique({
      ...query,
      where: args.where,
    }),
}),
```

---

## findMany

List with optional filters, ordering, and pagination. Use `refs.ModelFindManyArgs` and spread `args.input` into `findMany`.

```ts
users: t.prismaField({
  type: ['User'],
  args: {
    input: t.arg({ type: refs.UserFindManyArgs, required: false }),
  },
  resolve: (query, _root, args) =>
    prisma.user.findMany({
      ...query,
      ...(args.input ?? {}),
    }),
}),
```

All fields on `FindManyArgs` are optional. If you don't care about pagination, the client can omit `skip` and `take` and send only `where` and/or `orderBy` (or an empty object).

### findMany without pagination (where + orderBy only)

If you want a list field that does **not** expose `skip`/`take` at all, use `WhereInput` and `OrderByInput` as separate args instead of `FindManyArgs`:

```ts
users: t.prismaField({
  type: ['User'],
  args: {
    where: t.arg({ type: refs.UserWhereInputType, required: false }),
    orderBy: t.arg({ type: refs.UserOrderByInput, required: false }),
  },
  resolve: (query, _root, args) =>
    prisma.user.findMany({
      ...query,
      ...(args.where != null && { where: args.where }),
      ...(args.orderBy != null && { orderBy: args.orderBy }),
    }),
}),
```

This keeps the same filtering and ordering capabilities without exposing pagination in the API.
